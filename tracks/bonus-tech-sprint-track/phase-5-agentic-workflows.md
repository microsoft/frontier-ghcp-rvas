# Phase 5: Agentic Workflows

[Back to Technical Team Sprint Track](../bonus-tech-sprint-track.md)

**Duration:** 1.5 hours
**Focus:** Add GitHub Agentic Workflows to your deployed repository for continuous AI-driven maintenance

> This phase requires your code to be pushed to a GitHub repository with GitHub Actions enabled. Agentic workflows run as GitHub Actions workflows -- they cannot run locally.

Your application is deployed and working. Now you turn the repository into a self-maintaining project. GitHub Agentic Workflows let you describe automation in plain Markdown, and a coding agent executes it inside GitHub Actions with read-only permissions and safe outputs.

By the end of this phase, your repository will have at least three agentic workflows running on a schedule or triggered by events -- monitoring CI, improving code quality, and keeping documentation current.

## Before You Start

1. Make sure all code is pushed to a GitHub repository (not just local).
2. Install the `gh-aw` CLI extension:

   ```bash
   gh extension install github/gh-aw
   ```

3. Verify the installation:

   ```bash
   gh aw --help
   ```

4. Review the [GitHub Agentic Workflows documentation](https://github.github.com/gh-aw/) and the [Agentics gallery](https://github.com/githubnext/agentics) for available workflow templates.

## How Agentic Workflows Work

An agentic workflow is a Markdown file in `.github/workflows/` with two parts:

1. **Frontmatter** (YAML between `---` markers) -- defines the trigger, permissions, safe outputs, and tools
2. **Markdown body** -- describes the job in natural language

Example:

```markdown
---
on:
  schedule: daily

permissions:
  contents: read
  issues: read
  pull-requests: read

safe-outputs:
  create-issue:
    title-prefix: "[status] "
    labels: [report]

tools:
  github:
---

# Daily Repo Status

Create a daily status report for the team. Include recent activity,
open issues, PR status, and action items.
```

Run `gh aw compile` to generate the lock file, then commit both files.

## Sync Point: Review All Workflows (at ~1:00)

At the 1-hour mark, the team gathers to review each other's workflows:

1. Each person walks through their workflow Markdown (2 min each)
2. The DevOps engineer reviews permissions and safe-output scoping
3. The team triggers each workflow manually and checks the GitHub Actions output
4. Fix any issues found during review

## Verification

The team is done when:

- [ ] At least three agentic workflows are committed to `.github/workflows/`
- [ ] Each workflow has a corresponding `.lock.yml` file generated by `gh aw compile`
- [ ] Each workflow uses read-only permissions by default
- [ ] Safe outputs are scoped correctly (title prefixes, labels, max counts)
- [ ] At least one workflow has been triggered and produced visible output (an issue, a comment, or a report)
- [ ] The team can explain what each workflow does and when it runs
